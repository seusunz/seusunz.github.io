论文解读3-CoPE-VideoLM：用于高效视频语言模型的编解码基元
#VideoLM#Codec-aware Tokenization#视频压缩 2026-03-15

摘要

当前视频语言模型一个非常现实的瓶颈是：视频太长，而 RGB 帧太贵。 现有方法通常只能从长视频里抽取少量关键帧送入视觉编码器，这虽然能控制计算成本，却会损失时序细节；而如果把每一帧都当成完整图像去编码，Token 数量、预填充时延和显存压力又会迅速失控。

CoPE-VideoLM 的思路非常直接：既然视频本来就是以压缩格式存在的，那就不要无视编解码器已经提取出来的运动信息。作者把视频压缩中的运动矢量和残差直接作为建模对象，只对少量 I 帧做完整 RGB 编码，对大量 P 帧则使用轻量级的 Δ-Encoder 处理底层变化量。这一改动大幅降低了首字生成延迟和 Token 开销，同时在视频理解任务上保持甚至超过了现有方法。

核心贡献

  • 提出 codec-aware tokenization，直接利用视频压缩域中的运动矢量和残差，而不是只依赖 RGB 帧。

  • 设计轻量级 Δ-Encoder，将 P 帧变化量压缩成极少数 Δ-Tokens。

  • 提出对齐预训练策略，使压缩域特征能够与标准视觉编码器表征空间对齐。

  • 在多类视频理解任务上取得很强表现，同时显著降低 TTFT 与 Token 数量。

  • 为长视频、多模态实时交互和端侧 VideoLM 提供了更有工程可行性的路线。

1. 这篇论文在解决什么问题

VideoLM 的困难不只是“模型还不够强”，而是输入表示本身就很昂贵：

  • 视频包含大量重复帧与细粒度时序变化；
  • RGB 逐帧编码会造成严重计算冗余；
  • 上下文窗口有限，模型只能在“看得更长”和“看得更细”之间做痛苦取舍。

因此，很多方法采用固定抽帧，比如无论视频长短，只取 32 帧或 64 帧。这个做法虽然简单，但代价也很明显：

  • 宏观动作变化容易断裂；
  • 微观动作细节容易丢失；
  • 真正重要的时序信息可能根本没被采到。

CoPE-VideoLM 的核心判断是：视频中最值得传给模型的，不一定是完整 RGB，而是“发生了什么变化”。

2. 方法核心：把压缩域特征直接变成 Token

2.1 I 帧和 P 帧分工处理

作者没有完全放弃传统视觉编码器，而是做了有区分的设计：

  • I 帧：继续走完整 RGB 编码路径，保留高质量视觉语义；
  • P 帧：不再恢复为完整图像，而是直接处理运动矢量和残差。

这样做的本质，是把视频理解中的信息拆成两部分：

  • 静态语义由 I 帧提供；
  • 时序变化由 P 帧的压缩域基元提供。

2.2 Δ-Encoder 的角色

论文设计了一个轻量级的 Δ-Encoder，用来处理 P 帧变化量。它不需要像标准视觉编码器那样对整张图做高成本特征提取，而是只关注“变化本身”，最终把一帧或多帧变化压缩成极少量 Token。

这里最关键的收益有两个：

  • Token 数量急剧下降：P 帧只需要很少几个 Δ-Tokens；

  • 时延显著降低：大量帧不再经过重型图像编码链路。

这正是它能把首字延迟从秒级大幅拉低的根本原因。

2.3 两阶段训练保证语义对齐

如果直接把压缩域特征送进 LLM，一个自然问题是：这些特征和标准视觉表征能对齐吗？

作者用两阶段训练解决这个问题：

  • 第一阶段先让 Δ-Encoder 学会逼近常规视觉编码器提取的特征；

  • 第二阶段再把它接入基础 VideoLM 做端到端微调。

这样一来，模型既保留了压缩域表示的高效率，又尽量避免语义空间不一致带来的性能损失。

3. 实验结果说明了什么

论文在多类视频理解任务上验证了 CoPE-VideoLM，结果里最值得关注的不是单一分数，而是下面三点：

3.1 用更少 Token 换来更强时序覆盖

相比纯 RGB 抽帧方案，CoPE-VideoLM 用极少量 Δ-Tokens 就能表达更多视频帧的信息。这意味着模型不必在“多看几帧”和“控制上下文长度”之间二选一。

3.2 对强时序任务尤其有利

因为 P 帧本来就显式刻画了运动变化，模型在时序推理类任务上往往更有优势。对 VideoLM 来说，这比单纯增加关键帧数量更贴近问题本质。

3.3 系统效率提升非常实在

论文给出了非常有说服力的系统数据：在单张 RTX 4090 上，首字生成延迟（TTFT）从 2.39 秒 降到 0.33 秒，下降约 86%；同时 Token 使用量大幅减少。这说明 CoPE-VideoLM 不是“准确率略涨一点但系统没法落地”的工作，而是真正兼顾了模型效果与推理效率。

4. 为什么这篇工作值得重视

我觉得这篇论文最有价值的点，是它把 VideoLM 里一个长期被忽略的问题挑明了：

视频模型不应该只思考“怎么把更多 RGB 帧塞进模型”，而应该思考“视频里哪些信息根本没必要再走一遍昂贵的 RGB 编码”。

这其实是一种很典型的系统思维：

  • 不盲目堆输入；
  • 先看数据原生结构；
  • 再决定最便宜、最有效的信息表示方式。

从视频编解码角度看，这篇论文也很有启发：压缩链路里的运动矢量和残差，原本只是为了节省码率；到了 VideoLM 里，它们被重新解释成一种更适合长序列理解的语义载体。这种“Codec 到 Model”的迁移非常值得继续挖。

5. 优点与挑战

优点

  • 问题抓得准：不是继续在 RGB 特征融合上做边际优化，而是直接改输入表示方式。
  • 系统收益大：TTFT、Token 数量和长视频处理能力都明显改善。
  • 方法很有扩展性：适合长视频、实时交互、端侧部署等多个场景。
  • 跨领域融合自然：把视频压缩和多模态建模真正连接起来，而不是停留在概念借用层面。

挑战

  • 目前主要围绕 I / P 帧展开：对 B 帧及更复杂 GOP 结构的利用还不充分。
  • 压缩域表示仍较粗粒度：如果进一步下探到量化变换系数、块级运动信息，或许还能继续提效。
  • 融合策略还有优化空间：固定窗口下的 P 帧融合并不一定适合所有视频类型。
  • 工程依赖解码链路：不同视频格式、码流质量和解码实现都会影响实际可用性。

6. 前景展望

沿着这篇论文的思路，我觉得未来很可能会出现几条清晰路线：

  • 长视频理解会越来越依赖 codec-aware 表示，而不是继续暴力堆 RGB；
  • 实时视频助手与具身智能 会更需要这种低 TTFT 方案，因为交互系统对首字延迟非常敏感；
  • 端侧 VideoLM 会从中受益，因为 Token 和计算成本都被明显压低；
  • 压缩域与 Token 压缩技术融合 可能形成新一代高效视频输入管线。

从更大的视角看，CoPE-VideoLM 提醒我们：对于视频这种天然被压缩表示的数据，模型设计不应假装“压缩不存在”。谁能更好地利用压缩链路中的先验，谁就更有机会做出真正高效的视频大模型。
